이번 편에서는 3학년 1학기 웹 애플리케이션 팀 프로젝트로 개발했던 Campus OneClick에 대해 기술하였습니다.

학교에서 공부할 공간을 찾을 때 빈 좌석이 있는지, 사용할 수 있는 강의실이 있는지 확인하는 과정이 번거롭다고 느꼈습니다.
추가로 매 학기 시험 기간마다 좌석을 실제로 사용하지 않고 단순히 점유하기 위한 행동을 방지하고 싶었습니다.
그래서 우리 팀은 열람실 좌석 현황, 빈 강의실 조회, 예약 확인, 관리자 기능을 하나의 서비스 안에서 제공하는 것을 목표로 프로젝트를 진행하였습니다.

## 프로젝트 개요

- 프로젝트명: Campus OneClick
- 주제: 스마트 교내 유휴 공간 예약 및 탐색
- 주요 기능:
  - 열람실 좌석 현황 조회
  - 빈 강의실 조회
  - 공간 예약 기능
  - 내 예약 확인 기능
  - 관리자 전용 화면 제공

## 사용 기술

프론트엔드는 React를 사용하였고, 백엔드는 Spring Boot 기반의 REST API 서버로 구성하였습니다.
데이터베이스는 PostgreSQL을 사용하였으며, 백엔드에서는 Spring Data JPA를 통해 사용자, 강의실, 시간표, 예약, 좌석 관련 데이터를 관리하였습니다.

백엔드 주요 기술 스택은 다음과 같습니다.
- Java 21
- Spring Boot
- Spring Data JPA
- Hibernate
- Lombok
- Gradle
- PostgreSQL

프론트엔드에서는 백엔드 API를 호출하여 로그인 사용자 정보, 좌석 현황, 강의실 상태, 예약 정보 등을 화면에 표시하는 역할을 담당하였습니다.

프론트엔드에서는 페이지 이동, 상태 관리, API 호출, 조건부 렌더링 등을 주로 다루었습니다. 또한 로그인한 사용자의 권한에 따라 일반 사용자 화면과 관리자 화면이 다르게 보이도록 구현하였습니다.

## 제가 맡은 역할

이번 프로젝트에서 메인 페이지 구현을 담당하였습니다.
메인 페이지는 사용자가 서비스를 처음 접하는 화면이기 때문에, 전체 기능으로 자연스럽게 이동할 수 있도록 구성하는 것이 중요하였습니다.

제가 맡은 주요 기능은 다음과 같습니다.

1. 좌석 현황 조회 및 표시
2. 사용자 권한별 화면 제공
3. 내 예약 확인 모달 구현
4. 메인 페이지 UI 구성
5. 관리자 / 일반 사용자 화면 분기 처리

## 백엔드 API 연동

프론트엔드에서는 백엔드에서 제공하는 REST API를 활용해 필요한 데이터를 가져왔습니다.
대표적으로 사용자 인증 API, 좌석 조회 API, 강의실 정보 조회 API, 시간표 관련 API 등을 연동하였습니다.

백엔드에는 사용자 관리, 강의실 관리, 시간표 관리, 좌석 관리, 예약 관리 기능이 구분되어 있었습니다.

## 좌석 현황 조회 및 표시

좌석 현황은 /seats API를 호출하여 예약 가능한 좌석 수를 받아오는 방식으로 구현하였습니다.
백엔드에서 전달되는 좌석 데이터 구조를 확인한 뒤, 사용자가 이해하기 쉬운 형태로 화면에 표시하였습니다.

단순히 숫자만 보여주는 것이 아니라, 전체 좌석 수 대비 현재 사용 가능한 좌석 수와 점유율을 함께 보여주도록 구성하였습니다.
이를 통해 사용자는 현재 열람실이 얼마나 혼잡한지 빠르게 파악할 수 있습니다.

이 과정에서 프론트엔드는 화면만 만드는 것이 아니라, 백엔드에서 어떤 형태로 데이터가 전달되는지에 따라 상태값과 UI 구성이 달라진다는 것을 느꼈습니다.

## 사용자 권한별 화면 제공

이번 프로젝트에서는 일반 사용자와 관리자의 역할이 구분되어 있었습니다.
따라서 로그인 후 사용자의 권한에 따라 다른 화면을 보여주는 기능이 필요하였습니다.

백엔드의 사용자 데이터에는 학번, 이름, 학과, 이메일, Firebase UID, 권한 정보가 포함되어 있었습니다.
로그인과 회원가입 과정에서는 사용자의 인증 정보를 백엔드로 전달하고, 백엔드는 해당 정보를 검증한 뒤 사용자 정보를 반환하는 방식으로 구성되어 있었습니다.

프론트엔드에서는 백엔드로부터 전달받은 사용자 정보 중 role 값을 활용해 화면을 다르게 보여주었습니다.
role 값이 ADMIN인 경우 관리자 화면이 나타나도록 하고, 일반 사용자인 경우에는 실제 예약과 조회 기능을 사용할 수 있는 사용자 화면이 보이도록 구성하였습니다.

이 과정에서 sessionStorage에 저장된 값을 기준으로 조건부 렌더링을 적용하였습니다.
처음에는 관리자 계정으로 로그인해도 일반 사용자 화면처럼 연결되는 문제가 있었는데, 저장되는 role 값과 화면에서 비교하는 문자열이 일치하는지 확인하면서 문제를 해결하였습니다.

## 내 예약 확인 모달 구현

메인 페이지에는 사용자가 자신의 예약 정보를 확인할 수 있는 기능도 포함하였습니다.
"내 예약 확인하기" 버튼을 클릭하면 모달이 열리고, 사용자가 예약한 좌석 또는 강의실 정보를 확인할 수 있도록 하였습니다.

모달 방식으로 구현한 이유는 사용자가 다른 페이지로 이동하지 않고도 현재 화면에서 빠르게 예약 정보를 확인할 수 있기 때문입니다.
또한 예약 정보가 없는 경우에는 빈 화면이 아니라 안내 메시지를 보여주어 사용자가 현재 상태를 명확히 알 수 있도록 하였습니다.

이 기능을 구현하면서 조건부 렌더링을 많이 사용하였습니다.
예약 정보가 있는 경우와 없는 경우, 좌석 예약인지 강의실 예약인지에 따라 보여주는 내용이 달라져야 했기 때문입니다.

## UI 구성에서 고민한 점

메인 페이지는 여러 기능으로 이동하는 중심 화면이기 때문에, 너무 많은 정보를 한 화면에 넣으면 오히려 사용성이 떨어질 수 있다고 생각하였습니다.

특히 모바일 환경에서는 화면이 좁기 때문에 좌석 현황, 빈 강의실 조회, 예약 확인 기능을 카드 형태로 구분하고, 사용자가 필요한 기능으로 쉽게 이동할 수 있도록 구성하는 방향을 고민하였습니다.

또한 팀원들과 기능 배치에 대해 의견을 나누면서, 단순히 제가 맡은 화면만 구현하는 것이 아니라 전체 서비스 흐름 안에서 메인 페이지가 어떤 역할을 해야 하는지 생각하게 되었습니다.

## 배운 점

이번 프로젝트를 통해 React에서 상태 관리와 조건부 렌더링이 실제 서비스 구현에서 얼마나 자주 사용되는지 체감하였습니다.
또한 API 연동을 하면서 프론트엔드 개발자가 단순히 화면만 만드는 것이 아니라, 데이터 흐름과 서비스 구조도 함께 이해해야 한다는 것을 배웠습니다.

특히 사용자 권한에 따른 화면 분기, 예약 정보 표시, 모달 처리 같은 기능은 실제 웹 서비스에서도 자주 사용되는 구조라고 생각합니다.
이번 경험을 통해 React의 기본 개념을 실제 프로젝트에 적용해 볼 수 있었습니다.

또한 팀 프로젝트를 진행하면서 기능 구현뿐만 아니라 팀원과의 소통도 중요하다는 것을 느꼈습니다.
각자 맡은 기능이 다르더라도 결국 하나의 서비스로 연결되어야 하기 때문에, 데이터 구조나 화면 이동 방식에 대해 계속 의견을 맞춰야 했습니다.

## 마무리

Campus OneClick 프로젝트는 저에게 React와 웹 애플리케이션 개발 흐름을 실제로 경험해 볼 수 있었던 의미 있는 프로젝트였습니다.
특히 제가 맡은 메인 페이지는 사용자가 서비스를 처음 접하는 화면이기 때문에, 기능을 단순히 나열하는 것이 아니라 사용자가 쉽게 이해하고 이동할 수 있도록 구성하려고 노력하였습니다.

이번 프로젝트를 통해 프론트엔드 개발자로서 화면 구현 능력뿐만 아니라, 데이터 흐름을 이해하고 사용자 경험을 고민하는 태도가 중요하다는 것을 배웠습니다.
